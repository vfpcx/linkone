## 0. 为什么单列业务场景

技术场景测"某接口在某类输入下对不对"；业务场景测"**一个真实用户为了达成业务目的，走完整个旅程时，各种情况下系统对不对**"。同一个功能，技术场景绿 ≠ 业务旅程顺。本文用于**充分考虑用户实际使用的各种业务情况**，驱动 E2E 业务用例补全。

## 1. 买家（RT / 二批）旅程

| 编号 | 业务情况 | 期望 | 覆盖 | 证据/缺口 |
|---|---|---|---|---|
| B-RT-01 | 扫码进店→有货SKU→询价→(卖家确认)→成交 | 拿到询价单号，确认后自动出库、库存减 | ✅ | SELL-S1-01/02 E2E + curl 闭环 |
| B-RT-02 | **卖光了**：某SKU库存=0，买家进店 | 该SKU**不出现**在店里 | 🟡 | SF-S1-02(API)；❌ E2E 业务旅程 |
| B-RT-03 | **缺货下单**：询价量 > 当前库存 | 卖家确认时失败/提示缺货，不出库 | 🟡 | INQ-S5-01(API回滚)；❌ 买家侧旅程 |
| B-RT-04 | **一次买多个SKU**：同批发商多SKU加入询价 | 一张询价单含多明细，确认后逐一出库 | ❌ | 需确认 submitByRt items[] 多项行为 + E2E |
| B-RT-05 | **跨批发商**：店内多批发商，分别向不同家询价 | 各自独立询价单，互不串 | ❌ | 无 E2E；前端已限"一次单批发商" |
| B-RT-06 | **起批量**：下单量 < moqQty(起批量) | 前端拦截/提示达起批量 | ❌ | 前端有 stepper 上限=库存，起批量下限未测 |
| B-RT-07 | **空店**：新建仓、未上架任何SKU | 进店显示空态，不报错 | ❌ | 无 |
| B-RT-08 | **手机号非法**：提交询价填错手机号 | 前端拦截、不产生询价 | ✅ | SELL-S2-01 |

## 4. 覆盖小结与缺口

**已覆盖（业务旅程 E2E）**：B-RT-01/08、B-WA-01/03、B-EMP-01/03 —— 核心"卖货成交 + 幂等 + 员工入职"闭环。
**部分覆盖（仅 API 级）**：缺货回滚、并发抢货、越权、卖光不显示、员工码作废/用尽 —— 行为对，但缺"用户旅程"级验证。
**明确缺口（❌，建议补 E2E/业务测试）**：
1. B-RT-02 卖光后买家看不到（E2E）
2. B-RT-03 缺货下单旅程（买家提交→卖家确认失败）
3. B-RT-04 一次多SKU下单
4. B-RT-05 跨批发商分别询价
5. B-RT-06 起批量下限
6. B-RT-07 空店
7. B-WA-02 抢货整链业务级
8. B-EMP-02 员工离职后不能再注册（旅程）

**P2/暂不做**：B-WA-06 拒绝议价、B-EMP-04 多仓。
