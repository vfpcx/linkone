## 1. 为什么要场景测试

分层金字塔回答「在哪一层测」，场景测试回答「测哪些情形」。只测 happy path 的测试是**虚假绿灯**——上一轮 E2E 正是靠 S2/S4 场景（非法手机号、错误密码）和真实跳转，才暴露了 Bug B1/B2 路由与字段契约错位。

CLAUDE.md 对测试角色的硬性要求：
- UI 自动化测试：「区分正常流程、非法输入、边界场景、网络异常场景」
- Java 接口测试：「覆盖正常调用、非法参数、超时、重复请求、并发」

本文把这些固化为 **9 类标准场景（S1–S9）** + 标准化用例格式 + 覆盖矩阵。

| **S5** | 状态机非法转移 Invalid State | 在错误状态下触发动作 | 重复审批、已出库回退、已驳回再确认 | 单元 / 集成 |

## 6. 并发场景（S7）专项 · Java 21 虚拟线程

CLAUDE.md 要求高并发用 Java 21 虚拟线程。统一模板：

```java
@Test
@DisplayName("IV-S7-01 并发出库不超卖：100 虚拟线程抢同一 SKU")
void concurrentDeduct_noOversell() throws Exception {
    seedStock(skuId, 50);                 // 初始库存 50
    int threads = 100, qtyEach = 1;
    var latch = new CountDownLatch(threads);
    var ok = new AtomicInteger();
    try (var exec = Executors.newVirtualThreadPerTaskExecutor()) {
        for (int i = 0; i < threads; i++) {
            exec.submit(() -> {
                try { inventoryService.deductStock(ctx(skuId, qtyEach)); ok.incrementAndGet(); }
                catch (BizException ignore) { /* 库存不足正常 */ }
                finally { latch.countDown(); }
            });
        }
        latch.await(10, TimeUnit.SECONDS);
    }
    assertThat(ok.get()).isEqualTo(50);            // 恰好扣 50 次
    assertThat(currentStock(skuId)).isZero();      // 不超卖、不少扣
}
```

**S7 必测点**：库存并发扣减(不超卖)、批量调价并发(Redisson 锁)、邀请码 maxUses 并发消费、账单生成幂等键并发。

## 7. 依赖故障与超时（S8/S9）专项

| 场景 | 注入手段 | 预期 |
|---|---|---|
| Redis 断连(S8) | 测试中临时停 embedded-redis / 模拟 ConnectionException | 业务优雅降级，错误码语义化，不裸奔 90001（关联 Bug A 修复）|
| 短信 provider 失败(S8) | mock SmsProvider 抛异常 | 切备用 provider 或返回可重试错误 |
| 分布式锁等待超时(S9) | 占锁后并发请求 | 超时返回 43xxx 限频/繁忙，不死等 |
| 外部 ASR/地图超时(S9) | mock 延迟 > timeout | 兜底降级(本地 Haversine 距离等) |

> S8/S9 在 MVP 为**抽样覆盖**，重点放在「失败时错误码语义正确 + 不悬挂」，不追求全注入。
