## 2. PRD §2.1 锁定的 9 项决策

### ADR-001 · 坐标系最终选定（对应 Q-A01）

**上下文**：
PRD D36 倾向 GCJ-02。需要确认在数据库、地图 SDK、客户端 GPS、海外坐标之间如何统一。

**备选方案**：
- A. WGS84：国际标准，GPS 原生，海外友好；但国内主流地图（高德/百度/腾讯）默认接受 GCJ-02，每次都要转换
- B. GCJ-02：火星坐标系，国内地图直接显示无偏移；GPS 原始数据需转换
- C. BD-09：百度专用，与国内高德/腾讯不通用，不考虑

**决策**：**采用 GCJ-02 统一落库**。
- 落库字段：`lng DECIMAL(10,7), lat DECIMAL(10,7), coordinate_system VARCHAR(8) DEFAULT 'GCJ02'`
- 客户端 GPS 经 SDK 工具（前端 jsapi 自带 `convertFrom` 接口）转换为 GCJ-02 后再上传
- 第三方地图 SDK 默认输出 GCJ-02，直接落库
- 海外坐标（出海场景，v2 才考虑）：保留 `coordinate_system` 字段以便日后扩展为多坐标系并存
- 兜底场景：`source = raw_wgs` 时允许落库 WGS84，由后置任务批量转换

**理由**：
- 高德/腾讯 SDK 默认 GCJ-02，避免每次展示前转换
- 移动端 GPS 转换库（如 [eviltransform](https://github.com/googollee/eviltransform)）成熟稳定
- 数据库存 GCJ-02 后，距离计算（Haversine）误差与 WGS84 在国内均 < 0.1%，可忽略
- `coordinate_system` 字段为未来海外、港澳台留扩展空间

**后果**：
- 正：与地图 SDK 一致、展示无偏移；国内业务 0 转换成本
- 负：日后若要海外扩展需要批量转换或多坐标系并存；GPS 原始数据丢失（如取证需要）→ 通过 `source` 字段标识原始来源缓解

### ADR-002 · 地图 SDK 选型（对应 Q-A02）

**上下文**：
平台需要地图选点、距离计算、附近仓库推荐、收货地址逆向解析、地理围栏等能力。主要在 H5 + 小程序 + PC 端使用。

**备选方案**：
- A. 高德地图（阿里系）：B 端覆盖广、API 文档完善、配额价格透明、JS API 2.0 性能好；与阿里云 OSS/短信生态联动方便
- B. 腾讯地图：移动端微信生态体验更佳；B 端文档稍弱；与高德相比 POI 数据差距小
- C. 百度地图：BD-09 坐标系自闭环，与平台 GCJ-02 决策冲突；B 端市场份额下滑

**决策**：**主用高德地图 JS API 2.0 + Web 服务 API**。
- 前端：JS API 2.0（PC + H5），uni-app 用 `uni-amap` 插件
- 后端：高德 Web 服务 API（逆向地理编码、距离计算、IP 定位、行政区查询）
- 配额：试点期申请企业版（日 100w 次免费），费控由网关层做限频
- 容灾：保留腾讯地图作为二备（Service Provider 接口抽象，详见 02 模块文档）

**理由**：
- 高德 B 端商用方案成熟，与 OSS / 短信同主体计费简化对账
- JS API 2.0 移动端性能优于腾讯 H5
- GCJ-02 默认输出，与 ADR-001 一致

**后果**：
- 正：API 配额可控；与阿里云生态绑定降低对账成本
- 负：单一供应商风险（通过接口抽象 + 灰度切换缓解）

### ADR-003 · ASR 语音 SDK 选型（对应 Q-A03）

**上下文**：
PRD §10 + 01 §6 明确 MVP 仅普通话，三类单据（入库 / 出库 / 询价）支持语音录入；要求实时转写显示 + NLU 字段抽取 + 置信度反馈。

**备选方案**：
- A. 阿里云智能语音交互（NLS）：一句话识别（短语音 ≤60s）+ 实时语音识别（流式）+ Java SDK 完善；与 OSS 同账号
- B. 腾讯云 ASR：移动端体验好；价格略低；与微信生态接近
- C. 讯飞开放平台：方言识别业界领先（但 MVP 不需要）；价格偏高

**决策**：**阿里云 NLS 一句话识别 + 实时识别双通道**。
- H5 录音 ≤ 60s → 一句话识别（WebSocket 推流 + 后端转发）
- 实时转写界面 → 实时语音识别（流式，前端展示）
- NLU 字段抽取：在后端用规则引擎 + 正则 + 模糊匹配本商户 SKU 库；不依赖 ASR 提供商的 NLU
- 备份通道：腾讯云 ASR 作为灰度备份（接口抽象，详见 02 语音模块）
- 录音文件落 OSS，URL 写 `VoiceRecord` 表，30 天后自动删除

**理由**：
- 阿里云 NLS 短语音准确率 > 95%（普通话），满足 MVP
- 与 OSS 同主体计费 + 与 RocketMQ 同 VPC 减少跨网延迟
- NLU 自建可控（业务字段抽取规则随产品迭代灵活调整），避免 ASR 厂商 NLU 接口锁定

**后果**：
- 正：实时反馈 + 离线兜底（前端缓存 30 分钟）；自建 NLU 灵活
- 负：自建 NLU 维护成本（规则引擎需要随 SKU 库扩充）
- 计费：按调用次数预付费（详见 06 部署 §容量估算）

### ADR-004 · 短信通道选型（对应 Q-A04）

**上下文**：
短信用途：注册/登录/找回密码验证码、账单下发通知、代建单据通知、客诉通知等。日发送量预估试点期 5,000 条以内。

**备选方案**：
- A. 阿里云短信服务（dysmsapi）：API 稳定、模板审核快、与 OSS/NLS 同账号、价格 ¥0.045/条
- B. 腾讯云短信：与微信生态接近、价格相近
- C. 创蓝/云片等第三方：价格略便宜（¥0.038/条）但稳定性不如云厂商

**决策**：**主通道阿里云短信 + 腾讯云短信作灰度备份**。
- 主通道：阿里云 `dysmsapi`，签名「仓储云」+ 8 个模板（注册、登录、找回密码、换绑、账单下发、代建通知、客诉、强制清库）
- 备份：腾讯云短信，主通道连续 3 次失败自动切换（在 02 通知模块的 SmsProvider 抽象层实现）
- 限频：每手机号每场景 60s 不可重发，日上限 10 条（与产品规则 05 §16.1 对齐）
- 短信发送由 RocketMQ 异步触发，失败重试 3 次（间隔 10s/30s/2min）

**理由**：
- 阿里云模板审核流程快（24h 内通过）
- 与 OSS / NLS 同账号简化财务对账
- 双通道防止单一厂商故障导致登录/找回密码全链路瘫痪

**后果**：
- 正：高可用、对账简单
- 负：双通道维护成本；备份通道日常无流量需手动验活（建议每周自检脚本，详见 06 部署）

### ADR-005 · 对象存储选型（对应 Q-A05）

**上下文**：
用途：入库照片、撮合资料照片、账单导出 PDF/Excel、已收款凭证、语音录音、客诉附件、临期清库凭证。预估试点期月增量 50GB。

**备选方案**：
- A. 阿里云 OSS：与 ECS 同 VPC 走内网免流量费、CDN 联动、生命周期管理完善
- B. 腾讯云 COS：与微信小程序集成略佳；价格相近
- C. 七牛云：第三方独立厂商，价格便宜约 20%；与 ECS 跨网产生流量费

**决策**：**采用阿里云 OSS + 私有 Bucket + STS 临时令牌**。
- Bucket 划分：
  - `cangchu-cloud-prod-photos`（入库照片、清库凭证、客诉附件）→ 私有 + 永久
  - `cangchu-cloud-prod-voices`（语音录音）→ 私有 + 30 天 lifecycle 删除
  - `cangchu-cloud-prod-exports`（账单 PDF/Excel）→ 私有 + 7 天 lifecycle 删除
  - `cangchu-cloud-prod-public`（撮合资料、SPU 标准图）→ 公开读 + CDN
- 文件路径规范：`{tenant_id}/{wholesaler_id}/{module}/{yyyy}/{MM}/{uuid}.{ext}`
- 上传方式：
  - 小文件（< 5MB）：服务端代理转发 + 直传 OSS
  - 大文件（语音流）：STS 临时令牌前端直传
- URL 签名：私有 Bucket 访问通过签名 URL（30 分钟过期）

**理由**：
- 同 VPC 内网传输免流量费（试点期月省约 ¥800）
- Lifecycle 自动删除省手工运维
- 与 NLS / 短信 / ECS 同账号

**后果**：
- 正：成本可控、生命周期自动化、与生态绑定
- 负：单一供应商绑定（缓解：文件路径解耦，OSS Util 抽象层可日后切换）

### ADR-006 · 部署形态（对应 Q-A06）

**上下文**：
PRD D10 锁定模块化单体；试点 5 仓库 × 50 批发商 × 100 终端，并发预估 < 50 QPS。

**备选方案**：
- A. 单体单实例：成本最低；单点故障；不可热升级
- B. 单体多副本 + LB：试点期成本可控；具备灰度发布能力；故障容忍
- C. 微服务：与 D10 决策冲突，PASS

**决策**：**单体 JAR 多副本（≥2）+ Nginx 反向代理 + 共享 MySQL/Redis**。
- 应用层：2 副本起步，按 QPS 弹性扩容到 4 副本
- 状态：Session 落 Redis（Sa-Token 默认支持），副本无状态
- 部署：Docker 容器 + docker-compose（试点期）→ 后续切 K8s
- LB：Nginx 4 层（按 IP Hash 兜底，主要走 Sa-Token 无状态 Token）
- 详细架构图见 06 部署

**理由**：
- 试点期 QPS 低，2 副本足够；预留扩容路径
- 灰度发布能力（一副本升级一副本继续服务）
- 与 D10 模块化单体一致

**后果**：
- 正：故障容忍、灰度能力、成本可控
- 负：副本间共享 Redis 成为单点 → 部署 Redis Sentinel 主从

### ADR-007 · 多租户隔离方案（对应 Q-A07）★ 关键

**上下文**：
PRD 99 §2.1 倾向"共享库共享表 + tenant_id"。需评估隔离强度、性能、运维、未来扩展。

**备选方案**：
| 维度 | 共享库共享表（A） | 共享库分表（B） | 一租户一库（C） |
|---|---|---|---|
| 开发成本 | 低 | 中 | 高 |
| 隔离强度 | 弱（依赖 tenant_id 过滤） | 中（物理表隔离） | 强（数据库级） |
| 运维成本 | 低（统一备份/升级） | 中（表数量爆炸） | 高（N 个库管理） |
| 跨租户查询（OPS） | 易 | 难 | 难 |
| 性能 | 索引设计良好可承载 1000 租户 | 单表小，性能高 | 单库小，性能高 |
| 试点期适用 | ✅ | 过早优化 | 过早优化 |

**决策**：**采用 A 方案 — 共享库共享表 + tenant_id 行级隔离**。

**实施要求（强制）**：
1. **所有业务表必含 `tenant_id BIGINT NOT NULL`**（平台级表除外，详见 03 数据库 §共享表清单）
2. **所有业务表必有索引：`(tenant_id, primary_business_key)`**（如订单表 `(tenant_id, wholesaler_id, created_at)`）
3. **MyBatis 拦截器（TenantInterceptor）强制注入 `tenant_id`**：
   - SELECT：自动追加 `WHERE tenant_id = #{currentTenantId}`
   - INSERT：自动填充 `tenant_id`
   - UPDATE/DELETE：必须包含 `tenant_id` 条件，否则拦截抛 `PERMISSION_TENANT_LEAK_001`
4. **租户上下文（TenantContext）通过 Sa-Token 登录态注入**，在 ThreadLocal 中保持
5. **OPS 角色显式调用 `TenantContext.clearForGlobalQuery()` 才能跨租户查询**，并记录操作日志
6. **测试覆盖**：单元测试集成 `TenantLeakDetector`，CI 流水线必跑

**理由**：
- 试点期 5 仓库到 1000 仓库均可支撑，无需迁移
- 统一备份/升级/索引优化
- OPS 跨租户分析（账单汇总、客诉仲裁）便利
- 拦截器自动注入降低人为漏写 `tenant_id` 风险
- 业务表索引设计良好（详见 03 数据库 §索引策略）后性能不是瓶颈

**后果**：
- 正：开发成本最低；OPS 查询便利；扩展租户数无需迁移
- 负：依赖代码层（拦截器 + 测试）保证隔离强度，一次拦截器 bug 可能导致数据泄漏 → 通过 R-06 风险登记 + 强制测试覆盖缓解
- 长期演进：若单租户数据 > 5000w 行，按 `tenant_id` 做水平分表（迁移路径预留）

### ADR-008 · 月度账单生成时机（对应 Q-A08）

**上下文**：
PRD §1.4 + 04 §2.3 + S3：月 1 日 0:00 触发上月账单；试点 5 仓库 × 50 批发商 = 250 张账单，未来 1000 仓库 × 50 = 5w 张。同步全量在 0:00 完成压力大且非必要。

**备选方案**：
- A. 1 日 0:00 同步全量：简单；但容易 DB 抖动、失败重试成本高
- B. 异步队列 + 分批 + 重试：复杂度可控；失败自动重试；可观测
- C. 按需触发（ST 进账单页才生成）：用户体验差（第一次进慢）；并发难控

**决策**：**异步队列（RocketMQ / Redis Stream）+ 分批生成 + 三阶重试 + 幂等键**。

**实现方案**：
1. 1 日 0:00 触发调度任务（XXL-Job），发送"账单生成"任务到 RocketMQ topic `bill-generate`
2. 每条消息 = 一对（tenant_id, wholesaler_id, billing_month），幂等键 = `bill:{tenant_id}:{wholesaler_id}:{yyyyMM}`
3. 消费者并发数 = 10，按租户 hash 路由
4. 单条任务流程：
   - 查 DailySnapshot 聚合 → 生成 BillItem → 应用 BillingRule → 写 Bill → 标记完成
   - 单租户内事务 + 幂等键（INSERT IGNORE，重复执行返回已存在）
5. 失败重试：阶梯延时 10s/1min/10min，3 次后落 `bill_generate_failed` 表 + 告警 OPS
6. 全部完成后写 OperationLog + 推送 ST 站内信

**理由**：
- 解耦调度与执行，调度失败不影响业务可用性
- 幂等键防止重复生成
- 分批控制 DB 压力（5w 张账单分 100 批，每批 500 条，控制单 DB 写入压力）
- XXL-Job 提供完善 UI 监控失败任务

**后果**：
- 正：可观测、可重试、压力可控
- 负：依赖 MQ 中间件，部署成本略增

### ADR-009 · 容量公示刷新（对应 Q-A09）

**上下文**：
PRD §8.3：每 10 分钟刷新一次，前端展示最后一次成功快照。容量计算 = Σ(本店所有 in_stock > 0 的 SKU/Batch 件数)。

**备选方案**：
- A. 实时（每次写流水触发计算）：数据最新；写放大；高频出库时频繁重算
- B. 10 分钟快照：数据 10 分钟内可能偏差 1–2%；性能稳定
- C. 按需查询（每次 RT/WA 查容量时算）：实时；但读流量乘以 N 倍

**决策**：**10 分钟定时快照 + Redis 缓存 + TA 改设置时即时失效**。

**实现方案**：
1. XXL-Job 每 10 分钟执行 `CapacitySnapshotJob`：
   - 按 tenant_id 分组聚合：`SELECT tenant_id, SUM(in_stock_qty) AS used_qty, SUM(pallet_qty) AS used_pallet FROM inventory WHERE in_stock_qty > 0 GROUP BY tenant_id`
   - 计算剩余容量、利用率、模糊档位
   - 写 Redis：`capacity:tenant:{tenant_id}` 含 JSON（精确数 + 模糊档 + 时间戳），TTL 30 分钟
2. 查询接口：
   - 直读 Redis（命中率 > 99%）
   - Miss 时降级查 DB 实时算 + 异步回写缓存
3. TA 改可见性 / 改总容量 → 主动 `DEL capacity:tenant:{tenant_id}` 触发下次查询重算
4. 故障兜底：Redis 不可用时降级 DB 实时算（限频）

**理由**：
- 性能可控：每 10 分钟一次写入；读全走 Redis
- 数据偏差可接受：容量信息本身用于决策不需精确到件
- TA 改设置即时失效保证用户体验
- 与 PRD §8.3 一致

**后果**：
- 正：高性能、运维简单
- 负：极端高频写入场景（如试点期单店 1000 件/秒入库）数据延迟可能达 10 分钟 → 试点期不会出现

## 3. 架构师额外补充决策

### ADR-010 · 鉴权方案

**上下文**：
PRD 涉及 7 类角色、多角色兼任、多设备登录、Token 失效场景多（改密/换绑/封禁/退驻全部要踢出）。

**备选方案**：
- A. JWT 无状态：扩展性好；但 Token 失效控制困难（需要黑名单）
- B. Spring Security + Session：成熟；但与 H5 + uni-app 跨域 Session 复杂
- C. Sa-Token：国产开源，多账户体系（OPS / TENANT / WA / RT 可独立）、多设备会话、注解鉴权、踢人下线原生支持

**决策**：**采用 Sa-Token**。
- 多账户体系：`StpUtil`（默认 TENANT 体系）+ `StpOpsUtil` / `StpRtUtil` / `StpWaUtil`（独立 namespace）→ 满足"同手机号在不同角色域独立"
- 多设备登录：`StpUtil.login(userId, new SaLoginModel().setDevice("h5"))`
- Token 存储：Redis（与会话存储一致）
- 注解鉴权：`@SaCheckRole`、`@SaCheckPermission`、自定义 `@SaCheckTenant`

**理由**：
- 多账户体系天然契合 PRD 多角色架构
- 踢人下线 API 完善（改密/换绑/封禁场景直接 `StpUtil.kickout(userId)`）
- 中文社区活跃、文档完善

**后果**：
- 正：开发效率高、社区好
- 负：与海外标准（OAuth2）差异，未来出海需迁移

---

### ADR-011 · 主键策略

**上下文**：
分布式 ID 需求：单体多副本场景下自增 ID 会冲突；UUID 字符串太长影响索引；要求趋势递增便于范围查询。

**备选方案**：
- A. 数据库自增 ID：单体没问题，多副本同写同表无冲突（默认 InnoDB AUTO_INCREMENT）；但暴露业务量
- B. UUID v4：分布式安全；非递增导致 B+ 树插入跳跃；字符串占空间
- C. Snowflake（雪花算法）：64bit、趋势递增、含机器位、生成快

**决策**：**Snowflake ID（64bit）作为业务主键**。
- 实现：使用 Hutool 工具类 `IdUtil.getSnowflake(workerId, datacenterId)`
- 机器位：从配置中心读取（每个副本独立 workerId，避免冲突）
- 字段类型：`BIGINT UNSIGNED`
- 单据号（IN/OUT/IQ 等）与主键解耦：主键内部使用，单据号对外展示（详见 ADR-013）
- URL 中**不暴露主键**，使用单据号或 UUID 短码

**理由**：
- 趋势递增对 B+ 树插入友好
- 64bit 与 BIGINT 完全匹配
- 雪花算法在阿里、美团等大厂大规模验证

**后果**：
- 正：性能好、运维简单
- 负：依赖时钟同步（部署需 NTP 校时）；workerId 配置需小心

---

### ADR-012 · 软删除策略

**上下文**：
PRD §13 提及 RT 注销冷静期、WA 退驻 60 天恢复、员工禁用 30 天恢复等场景，硬删除不可行。

**备选方案**：
- A. 全表硬删除：数据不可恢复；外键引用断裂
- B. `is_deleted` BOOL + 索引：常见方案；查询需手动加条件
- C. `deleted_at` DATETIME + 全局拦截器：可追溯删除时间；MyBatis-Plus 原生支持

**决策**：**所有业务表统一 `deleted_at DATETIME DEFAULT NULL` + MyBatis-Plus 逻辑删除注解 + 全局 SQL 过滤**。
- 字段：`deleted_at DATETIME NULL`
- 逻辑：`deleted_at IS NULL` 表示未删除
- 配置：MyBatis-Plus `@TableLogic(value = "null", delval = "now()")`
- 索引：高频查询字段索引追加 `(tenant_id, status, deleted_at)`，避免 `deleted_at IS NULL` 全表扫
- 物理删除窗口：归档任务每月扫描 deleted_at > 1 年的数据 → 备份到 OSS 冷存 → 物理 DELETE

**理由**：
- 可追溯删除时间（满足 RT 注销冷静期、退驻 60 天等场景）
- MyBatis-Plus 原生支持，开发零成本
- 配合归档任务避免无限增长

**后果**：
- 正：数据安全、可恢复
- 负：表数据量增长 → 通过年度归档缓解

---

### ADR-013 · 单据号生成

**上下文**：
PRD §7.1 定义单据号格式 `IN-{tenant简码}-{YYYYMMDD}-{6位序列}`。Q-D10 询问：日切？月切？永不重置？

**备选方案**：
- A. 数据库自增表：原子性强；并发瓶颈
- B. Redis INCR + 日切重置：性能高；需要 Lua 保证原子
- C. UUID 短码：分布式友好；但不连续不便业务追溯

**决策**：**Redis INCR + Key 含日期前缀（自然过期）+ 简码缓存**。

**实现**：
1. tenant_simple_code 在 Tenant 表中预先生成（如 `CC01`），不可重复
2. 单据号生成：
   ```
   key = "doc_seq:{type}:{tenant_simple_code}:{yyyyMMdd}"  // 如 doc_seq:IN:CC01:20260520
   seq = INCR key
   EXPIRE key 86400 * 2  // 保留 2 天防跨夜边界
   doc_no = sprintf("%s-%s-%s-%06d", type, code, date, seq)
   ```
3. 日切自然重置（次日新 key，序列从 1 开始）
4. 账单号格式特殊：`BL-{tenant}-{wholesaler}-{yyyyMM}`（月切，由账单生成逻辑直接拼）
5. 盘点单 / 清库单 4 位序列同理

**理由**：
- 解决 Q-D10：日切（账单为月切）
- 序列长度足够：6 位 = 99w/天，远超试点期
- Redis 单线程 INCR 原子
- 日切自动避免序列号膨胀

**后果**：
- 正：高性能、可读性强、追溯方便
- 负：Redis 故障时降级到 DB 自增表（兜底，详见 06 部署）

### ADR-014 · 操作日志策略

**上下文**：
PRD 多处要求"操作日志强留痕"：OPS 代建、价格变动、批量调价、WK 代建出库、TA 仲裁等。审计要求 6 个月以上可查。

**备选方案**：
- A. 同步写 DB：实时；但增加请求延迟
- B. AOP 拦截 + 异步队列 + DB 批写：可控延迟；可观测
- C. 业务代码手写：灵活；易遗漏

**决策**：**AOP 注解（`@OperationLog`）+ 异步队列（Disruptor）+ MySQL 分区表 + OSS 归档**。

**实现**：
1. 注解定义：
   ```
   @OperationLog(module="PRICE", action="BATCH_UPDATE", desc="批量调价")
   public void batchUpdatePrice(...) {...}
   ```
2. AOP 切面拦截 → 收集（actor_user_id, actor_role, module, action, target_id, old_value, new_value, ip, ua, ts）→ 投递到 Disruptor RingBuffer
3. 消费者批量写 `operation_log` 表（按月分区，便于归档）
4. 6 个月后分区归档至 OSS（JSON.GZ），DB 中保留近 6 个月
5. 查询接口：近 6 个月走 DB，更早走 OSS（OPS 仲裁场景使用）

**理由**：
- AOP 减少业务代码侵入
- Disruptor 单线程高吞吐，0 锁
- 分区表便于按月归档

**后果**：
- 正：完整审计；性能影响 < 1ms
- 负：日志量大（试点期日均 10w 条，月 300w 条）→ 分区表 + OSS 归档解决
