## 1. 计费规则与公式

### 1.1 计费维度

平台支持 **件·天** 与 **托盘·天** 两种维度，按 TA 在 BillingRule 中的设置启用一种或两种。

| 维度 | 公式 | 单位 |
|---|---|---|
| 件·天 | `件数 × 在库天数` | piece·day |
| 托盘·天 | `占用托盘数 × 在库天数` | pallet·day |

计费维度**与批次开关无关**。批次开关只影响聚合粒度：

- 批次启用：按 (SKU, Batch) 计费，可下钻到批次明细
- 批次关闭：按 SKU 计费，账单明细只到 SKU 行

### 1.2 基准日规则

| 流水类型 | 计费起算日 | 计费截止日 |
|---|---|---|
| 入库 | 入库日的**次日 0:00** | 出库/退货/盘亏日**当天 23:59** |
| 盘盈 | 盘点单审批通过日（视同当日入库）的**次日 0:00** | 同上 |
| 盘亏 | — | 盘点单审批通过日**当天** |
| 退货 | — | 退货登记日**当天** |
| 出库 | — | 出库登记日**当天** |

> 设计意图：避免"当天入库当天出库"产生不可比的半天计费；起算日统一推到次日 0 点，保持每日完整 24h 计费颗粒。

### 1.3 单日计费明细

对某 (SKU, Batch?) 在某日 D：

```
是否在库 = (D >= 起算日) AND (D <= 截止日 OR 截止日为空)
当日件·天 = 件数 × (是否在库 ? 1 : 0)
当日托盘·天 = 占用托盘数 × (是否在库 ? 1 : 0)
当日费用(件) = 当日件·天 × 件·天单价
当日费用(托盘) = 当日托盘·天 × 托盘·天单价
当日费用合计 = 当日费用(件) + 当日费用(托盘)
```

批次启用时：每条 (SKU, Batch) 独立计算
批次关闭时：每条 SKU 直接计算（不分批次）

### 1.4 月度账单聚合

```
某 WA 当月账单应收金额 = Σ(每条计费实体每日费用合计)
                       + 折扣条目（负值）
                       + 减免条目（负值）
                       + 冲销条目（正负相抵的反向项）
```

> 调整条目必须可追溯到原条目；冲销不删原条目（详见 02 R10 / 04 §1.4）。

### 1.5 规则变更分段计费（R20）

当 TA 在月内修改单价/维度时：

```
当月费用 = Σ(变更日前每日费用 × 旧规则) + Σ(变更日及之后每日费用 × 新规则)
```

变更日的归属：**变更日按新规则计**（变更次秒生效）。

### 1.6 边界示例

| 场景 | 计算 |
|---|---|
| 5/10 入库 10 件，5/20 出库全部 | 起算 5/11 → 截止 5/20，10 天 × 10 件 = 100 件·天 |
| 5/10 入库，5/10 即出库（同日） | 起算 5/11 > 截止 5/10，**0 件·天**（不收费） |
| 5/10 入库 10 件，5/15 退货 4 件，5/25 出库 6 件 | (5/11~5/15) 10 件 × 5 天 + (5/16~5/25) 6 件 × 10 天 = 110 件·天 |
| 5/10 入库 10 件，5/20 盘亏 3 件 | (5/11~5/20) 10 件 × 10 天 + (5/21~) 7 件 × N 天 |
| 5/10 入库，5/15 盘盈 5 件 | 原 10 件按入库批次计；新 5 件起算 5/16 |

## 2. FIFO 拣货算法

### 2.1 批次启用时

```
def fifo_pick_with_batch(sku, want_qty):
    batches = query_batches(sku)
              .filter(in_stock > 0)
              .order_by(到效期 ASC, 入库时间 ASC, batch_id ASC)
    picked = []
    remain = want_qty
    for b in batches:
        if remain <= 0: break
        take = min(b.in_stock, remain)
        picked.append((b.batch_id, take))
        remain -= take
    if remain > 0:
        raise InventoryShortage(want=want_qty, available=want_qty - remain)
    return picked
```

排序优先级：到效期 → 入库时间 → batch_id

### 2.2 批次关闭时

```
def fifo_pick_without_batch(sku, want_qty):
    # 库存只有 SKU 维度，按"虚拟入库时间线"出库
    sku_inv = query_sku_inventory(sku)
    if sku_inv.in_stock < want_qty:
        raise InventoryShortage(want=want_qty, available=sku_inv.in_stock)
    # 简单扣减总数；展示给 WK 的 FIFO 提示退化为"先入库的批次先扣"
    # 入库时间线 = 该 SKU 的所有入库流水按时间排序，逐笔扣减直到满足 want_qty
    return [(None, want_qty)]
```

排序优先级：按入库时间 ASC

> 批次关闭时，库管员在出库页只看到"出库 N 件"的提示，不区分批次；系统在后台按入库时间顺序扣减库存总数。

### 2.3 混合状态（启→关 切换期）

切换为关闭后，**老批次保留独立库存状态**，按老 FIFO 顺序出完为止。出库算法变为：

### 2.4 指定批次出库（仅批次启用时可用）

WA 可在出库单上指定批次。系统行为：

| 指定情况 | 系统行为 |
|---|---|
| 指定批次在库充足 | 按指定批次扣减 |
| 指定批次缺货但其他批次有 | **提示** FIFO 替代方案，由 WK 选择是否替换；不自动替换 |
| 全部 SKU 缺货 | 拒绝出库（参见 04 §3.2） |

批次关闭时此功能隐藏。

### 2.5 临期批次提示（仅批次启用时）

当 FIFO 选中批次 < 临期阈值时：

- WK 端拣货页 hover 显示"临期 N 天"标签
- 不阻塞出库

### 2.6 部分出库与拆批

允许：单次出库单跨多个批次（拆批）。
- 批次启用时：按 FIFO 自动拆分，单条出库流水按批次粒度记录（一笔出库可能落多条流水）
- 批次关闭时：单条出库流水仅记录 SKU + 件数，不拆批

## 3. 库存与流水计算

### 3.1 库存数公式

批次启用时对 (SKU, Batch) 二元组：

```
当前在库件数 = 入库件数 + 盘盈件数 - 出库件数 - 退货件数 - 盘亏件数
当前占用托盘数 = 入库托盘数 + 盘盈托盘数 - 出库释放托盘数 - 退货释放托盘数 - 盘亏托盘数
```

批次关闭时对 SKU 一元组：

```
当前在库件数 = Σ(全部流水的件数变化)
当前占用托盘数 = Σ(全部流水的托盘变化)
```

### 3.2 流水类型枚举

| 类型 | 触发 | 影响 |
|---|---|---|
| 入库 | InboundRequest 入库登记 | 件 +N, 托盘 +M |
| 出库 | OutboundRequest 出库登记 | 件 -N, 托盘 按比例释放或手动设定 |
| 退货 | ReturnRequest 退货登记 | 件 -N, 托盘 按比例释放 |
| 盘盈 | CountSheet 审批通过 | 件 +N, 托盘 +M（可选填） |
| 盘亏 | CountSheet 审批通过 | 件 -N, 托盘 -M（同步释放） |
| 临期清库 | ExpiryClearance 审批通过 | 件 -N, 托盘 释放（仅批次启用时存在） |

### 3.3 托盘释放规则（简化版）

入库时记录"占用托盘数 M"作为整批属性。出库时：

```
释放托盘数 = ceil(占用托盘数 × 出库件数 / 入库件数)
```

> 即按"件数比例 + 向上取整"释放托盘。WK 在出库登记页可手动覆盖此值（应对实际拣货后托盘没腾空的情况）。

> 例：入库 100 件占 5 托盘，出库 40 件 → 默认释放 `ceil(5 × 40 / 100) = 2` 托盘；WK 可改为 1 或 0。

批次关闭时：以 SKU 维度按"近似入库时占用托盘均值"做反算（精度较低，作为提示值，WK 必须确认/覆盖）。

### 3.4 库存查询的可见性

| 查询者 | 可见范围 |
|---|---|
| TA / WK / ST | 本租户全部 SKU + [批次] + 流水 |
| WA / WE | 本商户全部 SKU + [批次] + 流水 |
| RT | 仅看 SKU 维度的"对外可见库存数"（受 WA "是否对外显示库存数"控制） |
| OPS | 默认不看明细；客诉仲裁场景下方可下钻 |

批次列在批次关闭时隐藏。

## 4. 状态校验与可达性

### 4.1 状态转换合法性

| 实体 | 不可达转换示例 |
|---|---|
| InboundRequest | 已驳回 → 已受理 ❌；已入库 → 已撤回 ❌ |
| OutboundRequest | 已出库 → 已撤回/已撤销 ❌；已撤销 → 已出库 ❌ |
| Inquiry | 已确认 → 已拒绝/已取消 ❌（只能走"已作废"R8） |
| Bill | 已结清 → 待核对 ❌（需先 R12 冲销已收款）|
| CountSheet | 已通过 → 任何状态 ❌（不可逆） |
| Wholesaler | 已退驻 → 已下架 ❌；已下架 → 正常 ❌（需走重新入驻） |

### 4.2 状态前置条件汇总

| 转换 | 前置 |
|---|---|
| 入库登记 | 当前用户=WK，单据=已受理，[若拍照开关=必填] ≥1 张照片，[若批次开关=启用] 批次号+保质期已填 |
| WK 现场代建入库 | 当前用户=WK，归属 WA ∈ 本店已入驻列表，[若拍照开关=必填] ≥1 张照片，[若批次开关=启用] 批次号+保质期已填；登记即同时生成入库流水 + 进入「待WA确认」队列 |
| WA 异议代建入库 | 当前用户=WA/被授权 WE，单据状态=待WA确认，必填异议理由；生效后即时反向冲销流水 |
| 72h 自动默认接受 | 入库时间 ≥ 72h 且状态=待WA确认 → 后台 worker 改状态=已确认+标记 auto_accept=true |
| WK 现场代建出库 | 当前用户=WK，归属 WA ∈ 本店已入驻列表，库存充足（不超过该 SKU 在库数），收货人+地址（含坐标）+提货时间已填，必经二次确认弹窗；若单笔件数 > 在库 50% 必经大额校验弹窗 |
| WA 对代建出库提客诉 | 当前用户=WA/被授权 WE，单据状态=已出库 且 source=wk_created，30 天内可提；提客诉不改库存/账单，仅状态转「客诉中」+ 进入 OPS 仲裁队列 |
| OPS 仲裁代建出库客诉 | 当前用户=OPS，单据状态=客诉中；仲裁结论可选「WK 责任」/「WA 责任」/「双方协商」/「无责」；任何结论都不改库存/账单，仅作为线下赔偿依据 |
| 出库登记 | 当前用户=WK，单据=已打印，FIFO 拣货成功 |
| 询价确认 | 当前用户=WA/被授权的 WE，单据=已提交 或 议价中（RT 已接受新价） |
| 自动生成出库单 | 询价确认成功的同一事务内（强一致） |
| 退驻通过 | 库存=0 + 账单全部"已结清" + 无"待受理/已打印/已确认"状态单据 |
| 账单冲销 | 任意状态均可（含已结清，参见 04 §3.4） |
| 已收款冲销 | 账单状态 ∈ {部分回款, 已结清} |
| 容量公示策略变更 | 当前用户=TA，1 分钟内 ≤ 3 次（防滥用） |
| 批次开关切换 | 当前用户=TA，24h 内 ≤ 2 次 |
| 拍照开关切换 | 当前用户=TA，无频次限制 |

## 5. 权限校验细则

### 5.1 校验顺序（每次请求）

```
1. 鉴权（token 有效）
2. 角色解析（actor_role，多角色时取请求 header 或默认主角色）
3. 租户/商户范围校验（资源 tenant_id / wholesaler_id 必须在用户的可访问集合内）
4. 角色权限校验（按 01 §4 矩阵）
5. 状态前置校验（参见 §4.2）
6. 业务规则校验（数量、金额、日期等）
```

任一步失败立即返回，不继续。

### 5.2 一人多岗权限并集

```
有效权限 = ⋃(user 所拥有的所有 role 的权限集合)
```

兼任优先级（决定 H5 端默认进入哪个工作台）：

```
TA > ST > WK > WA > WE
```

操作日志按**实际操作时使用的角色**记录：

```
log.actor_role = request.X-Actor-Role || user.primary_role
```

### 5.3 数据范围

| 角色 | 范围 |
|---|---|
| OPS | 全平台（业务明细按需下钻） |
| TA / WK / ST | tenant_id = 自己的租户 |
| WA / WE | wholesaler_id = 自己的商户，且 wholesaler.tenant_id = 入驻租户 |
| RT | 仅本人发起的 Inquiry + 本人的浏览历史 + 当前店铺公开数据 |

### 5.4 跨边界访问规则

- WA 不能查询其他 WA 的 SKU / 单据 / 库存 / 账单
- WK 不能查询本店内其他 WA 的销售额（但能看库存数，因为是仓内操作必须）
- ST 不能看库存明细（只看汇总到账单的件·天/托盘·天聚合数）→ 设计意图：账务与库存职责分离
  - 例外：兼任 WK 的 ST 通过 WK 身份切换可看

## 6. 数据隔离规则

### 6.1 软隔离（同库不同 tenant_id）

| 实体 | 隔离字段 |

| SKU / Batch / Inventory / StockMovement | wholesaler_id（间接含 tenant_id） |

| Inquiry | tenant_id + wholesaler_id + rt_user_id |

### 6.2 全局共享

| 实体 | 说明 |
|---|---|
| Platform / SPU / Blacklist / Announcement | 平台级，全部可见 |
| SpecType / SpecValue | 平台维护，全平台共享 |

- 一旦被 OPS 加入 Blacklist，全平台不可入驻

## 8. 容量公示算法

### 8.1 容量值定义

```
总容量(件) = TA 设置的固定值
当前占用(件) = Σ(本店所有 in_stock > 0 的 SKU/Batch 件数)
剩余容量(件) = max(0, 总容量(件) - 当前占用(件))
利用率(%) = round(当前占用(件) / 总容量(件) × 100)

总容量(托盘) / 当前占用(托盘) / 剩余容量(托盘) 同上
```

容量算法**与批次开关无关**：关闭时按 SKU 总数累加，启用时按批次累加，结果相同。

### 8.2 精度档位

| 档位 | 展示格式 |
|---|---|
| 精确 | `剩余 8,432 件 / 56 托盘` |
| 模糊 | `利用率 < 30%` / `30%–70%` / `> 70%`（三档） |

WA 端订阅"容量告警 < 20%"：基于**精确数据**判断（即使展示是模糊档），保证告警可靠。

### 8.3 刷新频率

- 后端每 10 分钟生成一次容量快照存缓存
- 前端展示该快照（含上次刷新时间）
- 故障时降级使用最后一次成功快照（不空白）

## 9. 临期 / 清库规则（仅批次启用时生效）

> 批次关闭时本章规则全部失效。临期相关入口在 UI 中隐藏。

### 9.1 临期判定

```
是否临期 = (到效期 - 当日) <= 临期阈值天数
```

临期阈值由 TA 在 BillingRule 中设置（默认 30 天）。

### 9.2 临期处理动作

| 角色 | 可做 |
|---|---|
| WK | 临期列表浏览、发送通知给 WA |
| WA | 收到通知后：降价处理 / 主动退货 / 不动作 |
| TA | 总览临期占比，决定是否调整接货策略 |
| 系统 | 每日 02:00 重算临期列表 |

### 9.3 强制清库（R19）

```
保质期 - 当日 <= 0  → 自动标"待清理"
WK 发起 ExpiryClearance 单（必附实物照片，不受拍照开关影响）
TA 审批通过
生成"出库"流水（类型=临期清库），件数 = 该批次剩余
计费截止 = 清库日当天
WA 收到通知 + 凭证
```

清库批次**不计入正常出库统计**，不影响 WA 销售数据，但仍按"在库期间"计入仓储费。

## 14b. 价格规则

### 14b.1 价格匹配优先级

RT 浏览或询价时取价顺序：

```
def resolve_price(rt_phone, sku, qty):
    if rt_phone is None:                      # 未登录
        return public_price(sku, qty)
    cp = customer_prices.find(rt_phone, sku)
    if cp and cp.is_active():                 # 命中专属价
        return cp.unit_price
    return public_price(sku, qty)             # 兜底公开价

def public_price(sku, qty):
    if qty >= sku.min_bulk_qty:               # 达到起批量
        return sku.bulk_price
    return sku.unit_price
```

`cp.is_active()` = 状态=生效 且 (过期时间 IS NULL OR 过期时间 > NOW())

### 14b.2 议价沉淀规则

| 决策点 | 规则 |
|---|---|
| 何时触发沉淀弹窗 | 议价确认 且 成交价 ≠ 公开价（按起批量判定的公开价） |
| 默认沉淀选项 | "永久专属价" 默认勾选 |
| 选项集 | 本次一次性 / 永久 / N 天（自定义 1-365） |
| 沉淀来源标记 | `source = from_inquiry`，含原询价单号 |
| 沉淀失败处理 | 不阻塞出库单生成；记错误日志触发 OPS 监控 |
| 已有专属价时 | 弹窗提示"该客户已有专属价 ¥X，是否覆盖？" |

### 14b.3 公开价变动联动

| 变动 | 联动 |
|---|---|
| WA 调整 SKU 公开价 | **不影响**任何已存在的 CustomerPrice |
| 价格管理页 | 对应 SKU 行显示「公开价变动 X% · N 条专属价仍生效 [批量复查]」 |
| 批量复查 | 列出所有该 SKU 的专属价 + 当前与公开价的差值，WA 可批量勾选跟随调整 |

### 14b.4 批量调价约束

| 项 | 规则 |
|---|---|
| 单次最多 | 500 条专属价 / 200 个 SKU 公开价 |
| 调价方式 | 涨/降 X% / 改为 ¥X / 加减 ¥X / 立即失效 / 设过期日 |
| 调价后限制 | 同一调价范围 5 分钟内不可再次批量调（防误操作） |
| 操作留痕 | 写 PriceChangeLog，含调前调后值、影响条目数、批次号、操作人 |
| 调价历史 | 保留 1 年；导出限 OPS / TA / WA 三类 |

### 14b.5 客户专属价的失效

| 触发 | 行为 |
|---|---|
| 过期时间 ≤ NOW() | 后台 worker 每日 02:00 标 `已过期` |
| WA 手动失效 | 状态变 `已失效`；RT 立即看公开价 |
| WA 删除 SKU | 该 SKU 下所有 CustomerPrice 一同标 `已失效` |
| WA 退驻 | 所有 CustomerPrice 标 `已失效`，但记录保留 60 天 |
| RT 注销账号 | 该手机号下所有 CustomerPrice 立即失效 |

### 14b.6 权限边界

| 角色 | 公开价 | 专属价 |
|---|---|---|
| WA | ✅ 增删改 | ✅ 增删改、批量调价 |
| WE | 🔵 授权时可改 | 🔵 授权时可改 |
| RT | 🔵 仅看匹配后单价 | 🔵 仅看匹配后单价 |
| TA / WK / ST / OPS | ❌ | ❌ |

### 14b.7 价格快照

每次出库单生成（不论手动或意向单自动转）必须记录"匹配时刻的单价快照"，写入 OutboundRequest 明细行，避免后续调价影响历史单据。

### 14b.8 整单优惠规则（仅意向单）

| 项 | 规则 |
|---|---|
| 优惠形式 | **仅支持「减 X 元」**（不支持百分比 / 调整总价 / 多项免费）|
| 金额约束 | `0 ≤ 优惠金额 ≤ 原总价 - 0.01` |
| 实付计算 | 实付 = 原总价 - 整单优惠金额 |
| 沉淀影响 | **不**触发客户专属价沉淀；不修改既有专属价 |
| 计费影响 | 整单优惠**仅作用于货款**，不影响仓储费（件·天/托盘·天）计算 |
| 议价中调整 | 议价回复时可改优惠金额；以"已确认"时刻为准 |
| 价格快照 | 出库单同步记录：原总价 + 整单优惠金额（分别落库）|
| 留痕 | 每次优惠金额变动写 OperationLog（操作人/原值/新值/备注）|
| 入口 | US-WA-06 询价处理确认前；US-WA-12 代下意向单创建时 |

### 14b.9 代下意向单约束

| 项 | 规则 |
|---|---|
| 触发角色 | WA 或被授权 WE |
| RT 身份 | 必填手机号；若未注册则自动创建 RT 账号（默认昵称 `用户****1234`，需 RT 后续可登录补完）|
| 状态生效 | 代下后**直接**进「已提交」（不需 RT 事后确认）|
| RT 通知 | 提交后短信 + 站内信通知 RT「您的代下订单 IQ-... 已提交」 |
| 字段记录 | `source = wa_proxy` + `proxy_creator_id` + 创建设备 IP |
| RT 端可见性 | RT 「我的意向单」展示该单（带"WA 代下"标识），可线下追问 / 提客诉 |
| 议价沿用 | 后续议价 / 确认 / 拒绝 / 沉淀专属价 均与普通意向单一致 |
| 代下 + 整单优惠 | 可在同一表单同时录入 |

### 14b.10 出库申请的 SKU 选择约束

| 场景 | 约束 |
|---|---|
| WA 创建出库申请的 SKU 下拉 | 仅展示本商户在该仓库 `in_stock > 0` 的 SKU |
| 下拉项展示 | "SKU 名称（可出库 N 件）" |
| 批次启用时的批次下拉 | 仅展示在库 > 0 的批次 |
| 语音识别命中无库存 SKU | 标红"该 SKU 当前无库存，请重选"；其他字段保留 |
| 件数 vs 在库校验 | 前端实时校验 + 后端提交时二次校验 |
| 库存数据延迟 | ≤10s（基于在库快照） |
| 意向单自动转出库 | 同样校验：若意向单提交时库存够、确认时不够 → 拒绝确认 + 通知 WA |
| 代下意向单（US-WA-12） | 同样校验：SKU 下拉只列 `in_stock > 0` 的 |
| 退货单 | 同样约束（仅在库 > 0 的批次可退）|

## 15. 地图与位置推荐规则

### 15.1 坐标系统一

平台**统一使用 GCJ-02 坐标系**（火星坐标），所有客户端写入/读取均以 GCJ-02 为准：

- GPS 原始 WGS84 → 入库前转换为 GCJ-02 落库
- 第三方地图 SDK（高德/腾讯）默认输出 GCJ-02，直接落库
- 海外坐标 / 港澳台特殊处理由架构师评估

### 15.2 Address 四元组

所有"地址"字段统一存为：

```
Address {
  text: string         // 用户可读地址
  lng: decimal(10,7)   // 经度
  lat: decimal(10,7)   // 纬度
  source: enum         // gps / map_pick / reverse_geocode / ip / manual
  precision: enum      // exact / city / region / coarse
  updated_at: datetime
}
```

### 15.3 推荐仓库算法（RT 进店）

```
def recommend_tenants(rt_location, sort_mode='distance'):
    candidates = tenants.filter(
        status = 正常,
        capacity_visibility != 不公开,
        coordinates IS NOT NULL
    )
    for t in candidates:
        t.distance = haversine(rt_location, t.coordinates)
    if sort_mode == 'distance':
        return candidates.sort_by(distance ASC)
    elif sort_mode == 'capacity':
        return candidates.sort_by(remaining_capacity DESC)
    elif sort_mode == 'composite':
        # 综合 = 距离权重 0.5 + 容量权重 0.3 + 上架活跃度 0.2
        return candidates.sort_by(composite_score DESC)
```

### 15.4 距离展示规则

| 距离区间 | 展示 |
|---|---|
| < 100 m | `< 100 m` |
| 100 m – 1 km | 显米，如 `850 m` |
| 1 km – 50 km | 显公里 1 位小数，如 `8.3 km` |
| > 50 km | 折叠到「远距离仓库」分区，按距离升序 |
| 坐标缺失 | 不显距离，标"位置未公开" |

### 15.5 位置精度兜底

| 来源 | 精度 | 用途 |
|---|---|---|
| GPS | exact (±10 m) | 全部场景 |
| 地图选点 | exact | 全部场景 |
| 反查地址 | city (~5 km) | 推荐排序可用，距离展示标注"约" |
| IP 定位 | region (~100 km) | 仅作为兜底初始位置 |
| 用户手选城市 | city | 推荐排序基于城市中心点 |

精度 ≥ region 时距离前缀显"约"。

### 15.6 隐私规则

- RT 当前位置**不持久化到 RT 个人资料**，仅在客户端本地缓存 7 天
- RT 收货地址（提交意向单时）属于业务数据，持久化到 Inquiry 实体
- 终端列表 / 排序结果**不包含其他 RT 的位置信息**
- OPS / TA / WA 不能查询其他角色的实时位置
- 涉及客诉仲裁时，OPS 经 RT 同意后方可下钻收货地址坐标

### 15.7 服务降级

| 场景 | 降级策略 |
|---|---|
| RT 拒绝定位授权 | 列表按 `composite` 排序，顶部提示"开启定位获得更准推荐" |
| 地图 SDK 加载失败 | 退化为纯文本地址输入；不计算距离；标"未获取坐标" |
| 坐标转换服务故障 | 按 WGS84 原始落库 + 标 `source=raw_wgs`，由架构师后置任务修复 |
| 第三方 ASR / 地图 API 配额超限 | 切换到二备供应商 / 降级到纯输入 |

### 15.8 收货地址校验（提交意向单时）

- 必须提供 lng/lat 才可提交（拒绝纯文本无坐标）
- 收货地址距仓库 > 100 km 时弹警告（"超远距离，请确认线下可送达"），用户确认后允许提交
- 与仓库坐标相同精度的地址才能参与距离计算

### 15.9 容量公示与位置耦合

- 容量公示策略 = 不公开 → 该仓库不出现在 RT/WA 推荐列表
- 容量公示策略 = 仅本店已入驻 → 在 WA「我的仓库」中出现；不在公开广场出现
- 容量公示策略 = 全平台 → 在公开广场 + RT 推荐 中均出现
